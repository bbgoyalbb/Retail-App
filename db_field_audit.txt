--------------------------------------------------------------------------------
NOTES ON ACCOUNTING RULES
--------------------------------------------------------------------------------

- Any received > 0 marks the section as "Settled" — no partial settlement concept.
- _pending can be negative (over-payment / credit) — never clamped or zeroed.
- _received stores the ACTUAL amount received, not capped to _amount.
- Settlement page balance uses pay_mode starts-with "Settled" as the filter,
  NOT the numeric pending value.
- Normalize fixes rounding drift ≤ ₹1. Repair fixes structural mismatches.
  Neither ever touches intentional over-payments (received > total + 0.01).

--------------------------------------------------------------------------------
ADVANCE RECORD TYPES (how to distinguish in the database)
--------------------------------------------------------------------------------

  Positive amount, mode = Cash/PhonePe/etc.  → genuine advance payment received
  Positive amount, mode = "Adjusted"         → normalize corrected a rounding issue
  Negative amount, mode = "Adjusted"         → advance consumed via settlement
                                               (amount = -advance_to_use)

--------------------------------------------------------------------------------
GAPS & NOTES — ADVANCES
--------------------------------------------------------------------------------

1. Excel export does NOT include the tally column for advances
   (adv_headers only has 5 columns: date, name, ref, amount, mode).
   FIXED in commit 26fc494 — tally is now exported and re-imported at row[5].

2. Backup restore preserves advances exactly as-is including tally state. ✓

3. Manual edit (PUT /advances/{id}) uses AdvanceUpdateRequest which supports:
   ref, name, amount, date, mode, tally — all optional. Any field not sent
   is excluded from the $set update (None = "don't touch").

4. There is no DELETE+recreate path for advances — delete is permanent.
   If an advance was consumed (negative Adjusted entry exists), deleting
   the original positive entry will leave an orphaned negative balance.
   FIXED — DELETE /advances/{id} now checks for an Adjusted entry on the
   same ref before deleting. Returns HTTP 400 with the consumed amount if
   a settlement adjustment exists, blocking the orphaned balance scenario.
